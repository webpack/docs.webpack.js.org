# [GSoC 2026] Project Proposal: Webpack's Documentation Redesign

## What?

<a href="https://github.com/webpack/webpack">webpack</a> is an industry-standard <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript">JavaScript</a> <a href="https://webpack.js.org/concepts/module-bundlers/">module bundler</a> used by millions of developers to build <a href="https://developer.mozilla.org/en-US/docs/Learn/Getting_started_with_the_web">modern web applications</a>. It plays a foundational role in today's frontend ecosystem and is commonly used in projects built with frameworks such as <a href="https://react.dev/">React</a>, <a href="https://vuejs.org/">Vue</a>, <a href="https://angular.io/">Angular</a>, and many others.

Despite webpack's continued relevance, active maintenance, and wide adoption, its primary user-facing surface, the <a href="https://webpack.js.org/">webpack website</a> and its accompanying documentation, has not evolved at the same pace as modern <a href="https://www.sphinx-doc.org/">documentation platforms</a>, <a href="https://designsystemsrepo.com/">design systems</a>, or current <a href="https://dx.tips/">developer experience</a> expectations.

This **<a href="https://summerofcode.withgoogle.com/">Google Summer of Code (GSoC) 2026</a>** project focuses on **modernizing webpack's website and documentation infrastructure** by:

* Automating API documentation directly from <a href="https://github.com/webpack/webpack">webpack's source code</a>
* Migrating the website to a modern, maintainable <a href="https://react.dev/learn/thinking-in-react">component-based architecture</a>
* Consolidating site content into a single, version-controlled codebase instead of relying on disparate runtime fetch scripts

The new documentation website will be built on top of **<a href="https://mintlify.com">Mintlify</a>**, a modern documentation platform that provides a seamless authoring experience, built-in search, and beautiful, customizable themes out of the box. Mintlify enables teams to create and maintain high-quality documentation with minimal overhead, making it an ideal foundation for webpack's redesigned documentation site.

This project is well suited for students interested in **<a href="https://github.com/topics/developer-tools">developer tooling</a>**, **<a href="https://github.com/topics/documentation">documentation systems</a>**, **<a href="https://frontendmasters.com/guides/architecture/">frontend architecture</a>**, and **large-scale open-source collaboration** within established ecosystems.

## Why?

The webpack core repository (<a href="https://github.com/webpack/webpack">webpack/webpack</a>) contains extensive inline documentation written using <a href="https://jsdoc.app/">JSDoc</a>. Core files such as <a href="https://github.com/webpack/webpack/blob/main/lib/Compiler.js">`Compiler.js`</a> alone contain **hundreds of documented APIs**, hooks, and internal abstractions.

However, much of this documentation is not currently exposed in a structured, searchable, or user-friendly format on the public documentation site.

In addition, the existing website (<a href="https://github.com/webpack/webpack.js.org">webpack/webpack.js.org</a>):

* Uses **<a href="https://react.dev/blog/2022/03/29/react-v18">React 17</a>**, which is now superseded by newer releases
* Renders documentation using **<a href="https://mdxjs.com/">MDX</a>** via custom <a href="https://webpack.js.org/concepts/loaders/">webpack loaders</a>
* Relies on a large number of bespoke UI components rather than a shared design system
* Is difficult to maintain, optimize, and evolve over time

Meanwhile, the <a href="https://github.com/nodejs/web-team">Node.js Website Infrastructure Team</a> (with which several webpack maintainers and mentors are involved) maintains a modern, reusable design system: <a href="https://www.npmjs.com/package/@node-core/ui-components">`@node-core/ui-components`</a>. This system is already used across official <a href="https://nodejs.org/">Node.js</a> web properties and is designed with **accessibility**, **consistency**, and **long-term sustainability** in mind.

Adopting this design system allows webpack to align visually and structurally with the broader <a href="https://openjsf.org/">OpenJS Foundation</a> ecosystem while significantly reducing maintenance overhead.

## Who?

This project is suitable for students comfortable working with modern web technologies and established open-source workflows.

Applicants should have experience with, or a strong willingness to learn:

* **<a href="https://react.dev/">React</a>** – component-based UI development for the web
* **<a href="https://jsdoc.app/">JSDoc</a>** – inline documentation for JavaScript APIs
* **<a href="https://www.markdownguide.org/">Markdown</a>** and **<a href="https://mdxjs.com/">MDX</a>** – Markdown extended with <a href="https://legacy.reactjs.org/docs/introducing-jsx.html">JSX</a>
* **<a href="https://tailwindcss.com/">Tailwind CSS</a>** – a utility-first CSS framework
* **<a href="https://git-scm.com/">Git</a>** and **<a href="https://github.com/">GitHub</a>** – version control and collaboration workflows
* **<a href="https://mintlify.com">Mintlify</a>** – the documentation platform used to build the new webpack documentation site. Mentees are expected to get familiar with Mintlify, including its <a href="https://www.mintlify.com/docs/customize/themes">theming capabilities</a> and <a href="https://www.mintlify.com/docs/customize/react-components">custom React components</a>

> [!TIP]
> Prior experience with webpack itself is **not required**. Curiosity about <a href="https://webpack.js.org/concepts/">build tools</a>, comfort reading large codebases, and an interest in developer experience will be particularly valuable.

  * <a href="https://www.mintlify.com/docs/customize/themes">Customizing themes</a> – learn how to configure and personalize the look and feel of the documentation site
  * <a href="https://www.mintlify.com/docs/customize/react-components">React components</a> – understand how to use and create custom React components within Mintlify
* Write **clean**, **maintainable**, and **well-documented** code

* Communicate blockers, design trade-offs, and decisions proactively
* Follow webpack's <a href="https://github.com/webpack/webpack/blob/main/CONTRIBUTING.md">contribution guidelines</a> and <a href="https://github.com/webpack/webpack/blob/main/CODE_OF_CONDUCT.md">code of conduct</a>

Progress will be evaluated based on:

* Quality and maintainability of code contributions

* Completion of agreed-upon milestones

## Tasks & Milestones

### Milestone 1: JSDoc → MDX API Generation

webpack's source code already contains extensive <a href="https://jsdoc.app/">JSDoc</a> annotations describing its internal APIs, hooks, and plugin interfaces.

The goal of this milestone is to build a **<a href="https://www.redhat.com/en/topics/devops/what-is-ci-cd">CI/CD</a> workflow** that:

* Extracts JSDoc comments from the <a href="https://github.com/webpack/webpack">webpack/webpack</a> repository
* Converts them into structured MDX documentation files
* Automatically opens pull requests against
  <a href="https://github.com/webpack/docs.webpack.js.org">webpack/docs.webpack.js.org</a> into a dedicated, versioned, folder, such as `docs/vX/api`

At this stage, the generated files do not need to be visually polished, they simply need to exist and be correct. Presentation improvements will follow in **Milestone 2**.

### Milestone 2: Design System Integration

The existing website relies heavily on custom UI components and legacy React patterns. This milestone focuses on integrating the shared <a href="https://www.npmjs.com/package/@node-core/ui-components">`@node-core/ui-components`</a> design system into the documentation repository.

The objective is to align webpack's documentation with the **Node.js design language**, improving accessibility, consistency, and maintainability. This includes styling the API documentation generated in **Milestone 1**.

> [!IMPORTANT]
> The Node.js design system is built on **<a href="https://tailwindcss.com/">Tailwind CSS</a>**. Applicants should be comfortable with <a href="https://tailwindcss.com/docs/utility-first">utility-first CSS</a> concepts.

### Milestone 3: Core Pages

With the design system in place, several key pages will be implemented or rewritten:

* **Home**

  * Introduces what webpack is, how it is used, and where it fits in the JavaScript ecosystem
  * Displays the current webpack release
  * Includes a prominent call-to-action linking to the **Download** page

* **Download**

  * Arguably the most important page on the site
  * Explains how to install webpack, official plugins, and related packages
  * <a href="https://docs.astro.build/en/install-and-setup/">Astro's install page</a> is a strong reference for structure and clarity

* **Guides**

  * Reorganize existing content from <a href="https://webpack.js.org/concepts/">webpack concepts</a>
  * Create a structured, progressive learning path for new users

* **About**

  * Consolidate governance, team, and project information
   (e.g. <a href="https://github.com/webpack/webpack.js.org/blob/main/src/utilities/fetch-governance.mjs">`fetch-governance.mjs`</a>)

> [!TIP]
> Students are **not expected to be professional technical writers**. Content may be reused from existing documentation or left for maintainers to draft. That being said, should a student wish to write, they are more than welcome to.

### Milestone 4: Expanding Automation

Once the main site is complete and stable, additional documentation sources can be automated, such as:

* Loader and plugin `README.md` files fetched from their repositories
  (similar to existing tooling like
  <a href="https://github.com/webpack/webpack.js.org/blob/main/src/utilities/fetch-package-readmes.mjs">`fetch-package-readmes.mjs`</a>)
* Normalizing and integrating this content into the documentation site

The overarching goal is to **minimize manual documentation maintenance** while improving accuracy and freshness.
